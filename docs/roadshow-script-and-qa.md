# 虾在忙路演演讲稿与评委问答

## 路演演讲稿

上周五下午三点，运营同学在群里发了一条消息：「这个学生的订单领取不到，辛苦帮忙看下」。十秒后，虾在忙自动识别问题、创建工单、通知负责人，五分钟内完成订单分配，并把结果回写到群里。这个过程，过去至少要等半天。

各位评委老师好，我的项目叫「虾在忙」。

它不是一个简单的聊天机器人，而是一个基于 OpenClaw 框架构建的飞书群支持 Agent。我们选择的第一个落地场景，是运营、师资、教务同学日常使用的中台小程序问题反馈群。

在这个群里，大家每天会提出很多真实业务问题，比如课程看不到、订单领取不到、课节需要删除、账号权限需要开通、回放异常等。过去这些问题都散落在群聊里，信息不结构化，谁处理、处理到哪一步、最后有没有解决，都很难持续追踪。

所以虾在忙要解决的问题很明确：把一句普通群消息，转成一个可以识别、可以跟进、可以回写状态、可以沉淀知识的处理链路。

第一页我们可以看到虾在忙的核心定位和运行数据。虾在忙从 4 月 22 日上线至今持续运行 16 天，已自动处理超过 134 个工单，服务了 50 多位来自运营、师资、教务等不同团队的同事，累计完成 48 次 API 自动操作。这些都是在真实支持群里跑出来的数据，不是测试环境。

第二页是产品演示。录屏展示的是虾在忙在真实支持群中的运行过程：业务同学在群里自然提问，虾在忙识别问题并关联处理工单，处理进展再同步回原会话。这个过程的关键不是让 AI 替人随便回答，而是让群聊里的问题进入一个可追踪的工作流。

第三页是架构。虾在忙基于 OpenClaw 构建。飞书群是入口，OpenClaw 作为编排中枢，连接意图识别、工单同步、审批门控、业务 API 和知识库。这样一来，群消息不再只是聊天记录，而可以被转化成结构化任务。OpenClaw 负责框架层的连接与执行，虾在忙负责业务场景里的判断规则、处理策略和回写逻辑。

第四页是安全控制。企业内部 Agent 最重要的不是"什么都自动做"，而是知道什么能自动、什么必须经过确认。比如 FAQ 回答、创建工单、查询状态、同步看板，可以直接处理；但涉及权限开通、订单分配、导师变更、课节课时调整，就必须经过审批或人工确认。虾在忙通过审批安全舱，把自动化控制在可审计、可回滚、可追踪的范围内。

最后一页是价值扩展。虾在忙的终点不是多一个群机器人，而是一套可以复制的内部 AI 工作流模板。它先在技术支持群里跑通闭环，后续可以扩展到更多高频支持场景，比如教务支持、运营支持、客户成功支持、内部 IT 支持等。核心价值是让问题少等待、让处理少打断、让过程可追踪、让经验可沉淀。

虾在忙要做的不是一个会聊天的机器人，而是让每一个群聊里的求助，都不会被遗忘。谢谢各位老师。

## 评委可能提问与回答案例

### 1. 你这个产品到底解决什么问题？

虾在忙解决的是企业内部支持群里"问题散、状态乱、经验难沉淀"的问题。业务同学在飞书群里自然提问，系统自动识别问题类型，创建或关联工单，推动处理流程，并把状态回写到群里。它不是单纯聊天，而是把群聊变成可追踪的工作流入口。

### 2. OpenClaw 在项目里具体承担什么作用？

OpenClaw 是底层 Agent 工作流框架，主要负责消息接入、流程编排、工具节点调用和执行记录。虾在忙是在 OpenClaw 之上构建的业务 Agent，负责中台小程序支持场景里的问题分类、工单规则、审批边界和回写策略。简单说，OpenClaw 提供通用框架，虾在忙提供具体业务能力。

### 3. 和普通 FAQ 机器人有什么区别？

普通 FAQ 机器人主要回答问题，解决的是"问答"场景。虾在忙解决的是"问题推进"场景。它不仅能回答，还能创建工单、同步状态、触发审批、调用业务接口、沉淀知识。它关注的是从问题提出到处理完成的完整闭环。

### 4. 为什么选择中台小程序支持群作为切入场景？

因为这个场景高频、真实、边界清晰。运营、师资、教务每天都会反馈课程、订单、权限、课节、回放等问题，历史样本足够多，问题类型也相对集中，非常适合先做 Agent 工作流验证。先从一个明确场景跑通，比一开始做通用助手更容易落地。

### 5. AI 判断错了怎么办？

我们不会让 AI 直接执行所有操作。系统会按风险分层处理：低风险问题可以自动回答或创建工单；中风险问题进入人工确认；高风险操作，比如权限、订单、课节变更，必须走审批。即使 AI 判断错误，也会被限制在可追踪、可人工接管的流程里。

### 6. 如何保证安全性？

安全设计有三层。第一是权限控制，不同操作绑定不同权限；第二是审批门控，高风险动作不能直接执行；第三是执行留痕，每一次识别、工单创建、状态回写、API 调用都有记录。我们追求的是可控自动化，而不是无限自动化。

### 7. 这个项目目前有什么数据支撑？

虾在忙从 4 月 22 日上线，至今已持续运行 16 天。在真实支持群中自动处理了 134 个工单，服务超过 50 位来自运营、师资、教务等团队的同事，累计完成 48 次 API 自动操作（包括订单分配、课节删除、课时修改等）。这些数据都来自线上真实流量，不是测试环境。

### 8. 你们的核心创新点是什么？

核心创新点不是单一模型能力，而是把 Agent 放进企业真实协作流程里。虾在忙把飞书群、工单、审批、业务 API、知识库串成闭环，让 AI 从"回答者"变成"流程协作员"。同时基于 OpenClaw 构建，具备更强的可复制性和扩展性。

### 9. 后续怎么扩展到其他业务场景？

我们会复用 OpenClaw 的框架层能力，替换业务策略层。比如中台小程序支持群跑通后，可以扩展到内部 IT 支持、客户成功支持、教务排课支持、运营数据支持等。每个新场景只需要定义问题分类、工具节点、审批规则和回写模板。

### 10. 如果没有足够知识库，系统还能用吗？

可以。知识库不足时，系统不会强行回答，而是优先创建工单或转人工。处理完成后，再把解决过程沉淀为知识。也就是说，虾在忙不是依赖一个完美知识库才能启动，而是在使用过程中持续积累知识。

### 11. 这个项目的商业价值怎么衡量？

可以从四个指标衡量：第一，问题首次响应时间是否降低；第二，重复问题人工处理次数是否减少；第三，工单状态是否更透明；第四，解决方案是否能持续沉淀复用。短期价值是提升支持效率，长期价值是形成组织级知识资产。

### 12. 为什么不用现成飞书机器人？

现成机器人通常解决消息触达和简单问答，缺少复杂业务流程编排能力。虾在忙的重点是基于 OpenClaw 把飞书群、工单、审批、业务 API 和知识库统一编排，形成可追踪闭环。这是从"机器人回复消息"到"Agent 推进流程"的区别。

### 13. 项目中遇到的最大技术挑战是什么？

最大的挑战是让 AI 在企业场景中"不出错"比"能做到"更重要。举个真实例子：虾在忙在连续处理两个不同学员的订单分配请求时，复用了上一个请求的订单 ID，导致分配到了错误的订单。这个问题的根因是上下文污染——模型把前一个请求的成功模式"记住了"，直接套用到了下一个请求。为此我们在 SKILL 指令中增加了"请求隔离"和"ID 来源校验"机制——每个请求必须独立查询，所有 ID 必须来自接口返回值，禁止编造或复用。这类安全约束的设计，是把 Agent 放到生产环境中才会遇到的真实挑战。

### 14. 上线以来有没有出过事故？你们是怎么处理的？

有，而且这恰恰是真实落地才会遇到的问题。上线第二周出现过一次订单分配的误操作——bot 调用了接口但使用了错误的订单 ID，接口返回了成功，但实际没有真正分配。我们当天排查定位了两个根因：一是 SKILL 指令文件超出了框架的上下文注入字符限制，导致关键的安全约束被静默截断，bot 根本没看到那些规则；二是缺少 ID 来源的强制校验。修复后我们做了三件事：调大上下文注入限制确保指令完整加载、增加请求隔离规则防止上下文交叉污染、加入执行前自检清单确保每个 API 调用都有合法的数据源。这次事故反而帮我们建立了更完善的安全机制。

### 15. 这个项目只有你一个人做的吗？投入了多少时间？

是的，从构思到上线大约两周时间。OpenClaw 框架本身提供了核心编排能力，我主要投入在业务策略层——定义问题分类规则、工单处理流程、审批边界和 API 对接。一个人能做到这个程度，也侧面说明基于 Agent 框架构建业务能力的效率是很高的。

### 16. 你的项目和市面上的 Coze / Dify / FastGPT 等平台有什么区别？

Coze、Dify 这类平台更偏向通用 AI 应用搭建，主要面向对话和内容生成场景，强调的是"搭建一个能聊天的 AI"。虾在忙基于 OpenClaw，更侧重企业内部协作流程编排——它不只是回答问题，而是把群聊消息转成工单、触发审批、调用业务 API、回写状态、沉淀知识。OpenClaw 的优势在于它原生支持飞书等 IM 渠道接入、内置记忆和工具调用机制、支持 Skill 热加载，更适合构建长期运行的企业级 Agent。这是从"AI 对话工具"到"AI 协作流程"的本质区别。
